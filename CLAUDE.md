# 學習進度管理 Web App — CLAUDE.md

## 專案目的

為12歲即將升國中的兒子建立**自主學習打卡系統**，家長可在低介入情況下追蹤學習狀況。

**核心目標：**

- 減少家長每天盯進度的負擔
- 建立孩子的自主學習習慣
- 訓練錯題診斷能力（不只記錄錯題，更記錄「為什麼錯」）
- 讓「閱讀習慣」可被執行與追蹤

**背景：** 不去補習，靠線上資源（均一、翰林雲端、ePop、Duolingo）自學。市面上沒有跨平台的學習行為打卡 + 家長輕量監控 + 錯題原因診斷工具，所以自建。

---

## 使用者角色

- **學生（國一兒子）**：每日打卡、登記錯題、查看自己的連續天數
- **家長（媽媽）**：查看週報、執行月底檢核

## 核心功能（MVP）

### 1. 每日打卡（學生端）

每日勾選完成項目，下列任務缺一不可：

| 項目         | 說明                       |
| ------------ | -------------------------- |
| 作業寫完     | 學校作業                   |
| 平台任務     | 均一 / 翰林雲端的今日任務  |
| 英文輸入     | ePop 或 Duolingo（擇一）   |
| 數學題庫     | 紙本隨堂評量 or 線上題庫   |
| 閱讀 20 分鐘 | 課外書（漫畫不算、課本不算） |

**打卡規則：**

- 每日打卡截止時間：當日 23:59
- 一週允許 1 次「免讀日」（學生自選使用）
- 閱讀項目沒完成 → 系統標記隔日「無螢幕時間」狀態（顯示給家長看，不強制阻擋）

### 2. 錯題登記（學生端）

每筆錯題包含：

- **科目**：國文 / 英文 / 數學 / 理化 / 社會
- **題目**：文字描述 or 拍照上傳（用 Supabase Storage）
- **錯誤原因**（下拉選單，**這欄最重要，不可省略**）：
  - 粗心
  - 不懂概念
  - 題意看不懂
  - 計算錯誤

> 設計重點：錯誤原因的分類比題目本身更重要。讓孩子養成「診斷錯誤」的習慣，這是自主學習能力的核心。

### 3. 家長週報儀表板

家長端一頁式顯示：

- 本週打卡率 %（5項任務全勾才算當日打卡成功）
- 本週各科錯題數量（長條圖）
- 各錯誤原因分布（圓餅圖，幫助診斷學習問題）
- 連續打卡天數
- **警示燈號**：
  - 綠：打卡率 ≥ 80%
  - 黃：70–80%
  - 紅：< 70%（提示需要介入）

### 4. 月底親子檢核頁

每月最後一週自動產生報告：

| 檢核項目       | 達標標準                         |
| -------------- | -------------------------------- |
| 作業準時交     | 月內「作業寫完」勾選率 ≥ 90%     |
| 錯題本有在記   | 月內至少登記 8 筆錯題            |
| 題庫練習有做   | 月內「數學題庫」勾選率 ≥ 80%     |

**判定邏輯：**

- 三項都達標 → 維持自主學習權（不需介入）
- 兩項以下達標 → 顯示「下個月需介入」提示

## 資料庫 Schema（Supabase）

完整 SQL 定義見 `supabase/schema.sql`。

**所有 table 都放在 `learning` schema**（不在預設的 `public`），原因是和 points-bank 共用同一個 Supabase 專案，必須分 namespace 避免衝突（特別是 `profiles` 表結構不同）。JS client 在 `src/lib/supabase/client.js` 和 `server.js` 已透過 `db: { schema: 'learning' }` 設成預設,所以 source code 裡 `.from('profiles')` 可以照常寫,不用加前綴。

> Supabase Dashboard 上要記得到 **Settings → API → Exposed schemas** 把 `learning` 加進去,PostgREST 才會公開。

### `learning.profiles`

```sql
id              uuid PK (auth.users.id)
role            text  -- 'student' | 'parent'
display_name    text
created_at      timestamptz
```

### `learning.daily_checkins`

```sql
id                    uuid PK
user_id               uuid FK -> learning.profiles.id
date                  date
homework_done         bool
platform_task_done    bool
english_input_done    bool
math_practice_done    bool
reading_done          bool
is_rest_day           bool  -- 該週的免讀日
created_at            timestamptz
updated_at            timestamptz

UNIQUE(user_id, date)
```

### `learning.mistakes`

```sql
id            uuid PK
user_id       uuid FK -> learning.profiles.id
subject       text  -- '國文' | '英文' | '數學' | '理化' | '社會'
description   text
image_url     text  -- Supabase Storage URL, nullable
reason        text  -- '粗心' | '不懂概念' | '題意看不懂' | '計算錯誤'
created_at    timestamptz
reviewed_at   timestamptz  -- 預留給未來「複習提醒」功能
```

### `learning.monthly_reviews`（autogenerated）

```sql
id                          uuid PK
user_id                     uuid FK -> learning.profiles.id
month                       date  -- 月份第一天
homework_completion_rate    numeric
mistake_log_count           int
practice_completion_rate    numeric
passed                      bool  -- 三項是否都達標
generated_at                timestamptz
```

### RLS 政策

- 學生只能讀寫自己的 `daily_checkins` 和 `mistakes`
- 家長可讀取所有學生的資料（之後可能加二兒子，預留多孩支援）
- 家長無法寫入學生的打卡記錄（避免代填）

## UI / UX 重點

1. **學生端要簡單，避免「學習平台感」**——他已經有均一和翰林雲端，這個工具的價值是「快速勾選＋輸入錯題」，不是內容平台
2. **打卡用大按鈕**，一鍵勾選，不要彈窗、不要二次確認
3. **連續打卡天數要視覺化**——用火焰 icon 或計數器，遊戲化機制是這個年齡的有效誘因
4. **家長端一頁式儀表板**，掃一眼就掌握全貌，不需要點來點去
5. **手機優先**（學生用手機打卡較多，家長也常用手機看）
6. **錯題拍照流程要快**——點「+」→ 拍照 → 選科目 → 選原因 → 完成（4 步以內）

## 與 points-bank 的關係

本專案已從 points-bank monorepo 拆出來,放在獨立的 repo `evelynytliu/learning-tracker`,但**和 points-bank 共用同一個 Supabase 專案**。為避免 table 名稱衝突(特別是 `profiles` 表),本專案所有 table 都放在 `learning` schema,points-bank 的 table 在 `public` schema。

**未來串接構想：**

- Supabase 已共用,兩個 app 連同一個 DB
- 在 `learning.daily_checkins` insert 達標 / `learning.monthly_reviews` 通過時,呼叫 points-bank 在 `public` schema 的點數發放 RPC
- 或反過來:points-bank 的家庭管理員身分可直接讀本系統的儀表板(跨 schema query)

短期內**不需要**做整合，先把學習追蹤 MVP 做穩。

## 備註

- 這是家用內部工具，不需要設計給陌生使用者，UI 可以針對「兒子 + 媽媽」兩個人最佳化
